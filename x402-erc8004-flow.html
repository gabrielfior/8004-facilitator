Phase 1: Payment (2 paths)

1
Client → Resource Server
GET /weather. Server returns 402 + two payment options: USDC $0.01 (EIP-3009) or DAI $0.01 (Permit2). Client picks based on which token they hold.
2
Client → Facilitator
Path A (USDC): signs TransferWithAuthorization (EIP-3009). Path B (DAI): signs PermitWitnessTransferFrom (Permit2) with spender=x402 Permit2 Proxy. Sends POST /verify.
3
Client → Resource Server
Retries GET /weather with PAYMENT-SIGNATURE header.
4
Resource Server → Facilitator → On-chain
Server calls POST /settle. Facilitator routes to transferWithAuthorization (USDC) or x402 Permit2 Proxy settle (DAI). Client must have approved Permit2 contract for DAI beforehand.
5
Resource Server → Client
200 OK + settlement tx hash. Server does NOT have Agent's private key.

Phase 2: Proof & Reputation

6
Client → Agent signing endpoint
POST /sign-proof { settlementTxHash, reqBody, respBody }. Agent verifies settlement on-chain, signs keccak256(agentId, req, resp) with own key. Returns X-Reputation-Proof.
7
Client → On-chain (single tx)
FeedbackGateway.markUsed(hash) → ReputationRegistry.giveFeedback(agentId, 95) → unmarkUsed on failure.
